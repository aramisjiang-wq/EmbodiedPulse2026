## 📋 核心原则

### 1. 文档命名规范（强制要求）

**⚠️ 核心原则**：
- **所有文档名称必须包含日期**
- **文档更新时，必须更新文档名称中的日期**
- **这是AI的强制要求，必须时刻遵守**

#### 命名格式
- **标准格式**：`文档名称_YYYYMMDD.md`
- **日期格式**：`YYYYMMDD`（8位数字，如：20251212）
- **示例**：
  - `Star功能方案_20251212.md`
  - `AI开发原则与规范_20251212.md`
  - `部署完整指南_20251212.md`

#### 文档更新规则
- **文档更新时，必须重命名文件，更新日期部分**
- **旧文档保留**：如果文档有历史价值，保留旧版本
- **新文档创建**：更新时创建新日期的文档，或重命名现有文档

#### AI操作要求
- **创建新文档时**：必须包含当前日期
- **更新现有文档时**：必须更新文档名称中的日期
- **检查文档时**：发现没有日期的文档，自动添加日期
- **这是AI的默认操作习惯，无需用户提醒**

### 2. 文档分类规范

#### 文件夹编号系统

所有文档按产品开发顺序编号（01-12）：

| 编号 | 文件夹名 | 阶段 | 说明 |
|------|---------|------|------|
| 01 | 需求文档 | 需求分析阶段 | 产品需求文档（PRD）、功能需求分析 |
| 02 | 产品文档 | 产品设计阶段 | 产品介绍、设计理念、开发过程分享 |
| 03 | 技术文档 | 技术设计阶段 | 技术架构、数据库设计、技术方案 |
| 04 | 功能说明 | 功能开发阶段 | 功能详细说明（按模块分类） |
| 05 | 开发记录 | 开发过程阶段 | 开发过程记录、复盘、建议 |
| 06 | 部署上线 | 部署上线阶段 | **所有部署相关文档**（部署方案、自动部署、故障修复等） |
| 07 | 优化记录 | 优化迭代阶段 | UI/UX优化和性能优化记录 |
| 08 | 测试文档 | 测试验证阶段 | 测试结果和功能检查 |
| 09 | 问题修复 | 问题解决阶段 | Bug修复记录（按问题类型分类） |
| 10 | 使用说明 | 用户使用阶段 | 用户使用指南（不含部署相关） |
| 11 | 开发规范 | 规范制定阶段 | 开发原则、规范、最佳实践 |
| 12 | 沟通记录 | 沟通交流阶段 | 开发沟通历程 |

#### 文档归类原则

1. **部署上线（06）**：
   - 所有部署相关文档
   - 包括：部署方案、自动部署、手动部署、故障修复、服务器配置等
   - **目的**：方便查找部署相关文档

2. **功能说明（04）**：
   - 按功能模块分类
   - 每个功能模块有独立子文件夹
   - 包括：论文抓取、新闻功能、招聘信息等

3. **问题修复（09）**：
   - 按问题类型分类（UI问题、功能问题、数据问题等）
   - 每个问题类型有独立子文件夹

4. **优化记录（07）**：
   - 按优化类型分类（UI优化、功能优化、性能优化等）
   - 每个优化类型有独立子文件夹

### 3. 文档合并规范

#### 合并原则

**相同目的的文档应合并**，避免文档分散：

1. **部署相关文档**：
   - 合并为：`部署完整指南_YYYYMMDD.md`
   - 包含：自动部署、手动部署、故障修复等所有内容

2. **文档整理记录**：
   - 合并为：`文档整理总结_合并_YYYYMMDD.md`
   - 保留历史记录，但合并到单一文档

3. **沟通记录**：
   - 合并为：`开发沟通记录_合并_YYYYMMDD.md`
   - 按日期汇总，保留历史记录

#### 合并时机

- 当同一目的的文档超过3个时，考虑合并
- 当文档内容重复度超过50%时，必须合并
- 合并时保留重要信息，删除重复内容

### 4. 文档更新规范

#### 功能改动时的文档更新清单（必须执行）

**当涉及产品功能改动时（新增、修改、删除功能），必须同步更新：**

1. **PRD文档**（必须）
   - 文件位置：`docs/项目文档/01-需求文档/PRD_产品需求文档_YYYYMMDD.md`
   - 更新内容：功能需求描述、功能优先级、用户场景、功能规格说明

2. **SPEC文档**（必须）
   - 文件位置：`docs/项目文档/03-技术文档/SPEC_技术规格文档_YYYYMMDD.md`
   - 更新内容：技术实现方案、API接口规范、数据库设计变更

3. **CHANGELOG文档**（必须）
   - 文件位置：`docs/项目文档/05-开发记录/发版记录_CHANGELOG.md`
   - 更新内容：版本号、更新日期、功能变更说明

4. **README文档**（必须）
   - 文件位置：`docs/项目文档/README.md`
   - 更新内容：文档索引、新增文档链接

5. **相关功能说明文档**（必须）
   - 文件位置：`docs/项目文档/04-功能说明/`
   - 更新内容：功能详细说明、使用指南

### 5. 文档索引规范

#### README.md维护

- **位置**：`docs/项目文档/README.md`
- **作用**：文档索引，快速查找文档
- **更新要求**：
  - 新增文档时，必须更新README.md
  - 文档重命名时，必须更新README.md中的链接
  - 文档删除时，必须从README.md中移除

#### 文档链接规范

- 使用相对路径：`./06-部署上线/部署完整指南_20251212.md`
- 链接必须指向最新版本的文档
- 文档重命名后，必须更新所有引用链接
